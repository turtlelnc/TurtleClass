# RFC-004：Ed25519 设备凭证格式

## 状态

Accepted

## 背景

Phase 2 需要实现正式的设备认证系统。当前设备注册仅基于内存，没有密码学认证。需要定义设备凭证格式和签名机制。

## 问题

如何选择设备凭证格式和签名算法？

## 目标

- 强安全性（防伪造、防篡改）
- 轻量级（适合嵌入式/低性能设备）
- 易于实现和维护
- 支持密钥轮换

## 非目标

- 复杂的 PKI 基础设施
- 硬件安全模块（HSM）
- 多因素认证（Phase 3 考虑）

## 提案

**决定：使用 Ed25519 作为设备签名算法。**

### 密钥生成

1. 设备首次启动时生成 Ed25519 密钥对
2. 私钥存储在设备本地安全位置
3. 公钥提交给服务器进行注册

### 凭证格式

```json
{
  "device_id": "<uuid>",
  "display_name": "Classroom PC",
  "public_key_hex": "<ed25519-public-key-hex>",
  "registered_at_unix": 1721400000,
  "signature": "<server-signature>"
}
```

### 签名流程

**上传事件时：**

1. 客户端序列化事件组为规范 JSON
2. 计算 JSON 的 SHA-256 哈希
3. 使用私钥对哈希进行 Ed25519 签名
4. 将签名附加到请求头 `X-Signature: <hex>`

**服务器验证：**

1. 从请求中提取事件组和签名
2. 使用相同方法计算事件组哈希
3. 使用存储的公钥验证签名
4. 验证失败则拒绝上传

### 规范序列化

为确保签名一致性，事件组必须序列化为规范 JSON：

- 键按字母顺序排序
- 无多余空格
- UTF-8 编码
- Unix 时间戳（非 ISO 字符串）

示例：

```json
{"class_id":"hex...","events":[{"badge_delta":0,"class_id":"hex...","compensates_event_id":null,...}],"group_id":"hex..."}
```

### 重放攻击防护

1. **设备本地序号**：每个事件包含递增的 device_local_sequence
2. **服务端检查**：拒绝序号不大于上次接受的事件
3. **时间窗口**：可选地拒绝超过一定时间的事件

### 密钥存储

**Windows 客户端：**
- 私钥存储在 `%APPDATA%\TurtleClass\keys\`
- 文件权限设置为仅当前用户可读写

**macOS 服务端：**
- 私钥存储在 `~/Library/Application Support/TurtleClass/keys/`
- 文件权限设置为 0600

### 密钥轮换

1. 生成新密钥对
2. 用旧密钥签名轮换请求
3. 服务器更新公钥记录
4. 旧密钥保留一段时间用于验证历史事件

## 数据与兼容性影响

- 现有设备需要重新注册获取新凭证
- 事件日志增加签名字段

## 安全影响

- 私钥泄露意味着设备可被伪造
- 必须实施密钥安全存储
- 定期轮换降低泄露风险

## 测试计划

1. 验证签名生成和验证
2. 验证篡改检测
3. 验证重放攻击防护
4. 验证密钥轮换流程

## 迁移计划

从当前无认证迁移：

1. 服务器进入维护模式
2. 所有设备生成新密钥对
3. 批量注册新设备
4. 启用签名验证
5. 退出维护模式

## 替代方案

1. **RSA-2048**：密钥太大，签名慢
2. **ECDSA-P256**：实现复杂，易出错
3. **HMAC**：需要预共享密钥，不适合多设备

## 未解决问题

- 是否需要硬件绑定（如 TPM）
- 密钥备份策略

## 决策

使用 Ed25519 进行设备认证，规范 JSON 序列化确保签名一致性，设备本地序号防止重放攻击。
