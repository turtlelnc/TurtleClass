# RFC-003：管理员界面与权限

## 状态

Accepted

## 背景

Phase 2 需要实现管理员账号和权限系统。需要确定管理员界面的位置和使用方式。

## 问题

管理员界面应该放在哪里？如何与普通用户界面区分？

## 目标

- 保持单一程序部署简单性
- 清晰区分普通用户和管理员权限
- 防止未授权访问管理功能
- 便于审计管理员操作

## 非目标

- 独立的管理程序
- Web 管理界面（Phase 2）
- 复杂的多角色 UI

## 提案

**决定：在同一个程序中通过管理员密码授权进入管理入口。**

### 权限模型

```
┌─────────────────────────────────────┐
│          TurtleClass 程序            │
├─────────────────────────────────────┤
│  普通模式 (默认)                     │
│  - 查看学生                          │
│  - 加减分                            │
│  - 使用模板                          │
│  - 撤销自己的操作                    │
└─────────────────────────────────────┘
                  ↑
         输入管理员密码
                  ↓
┌─────────────────────────────────────┐
│  管理员模式 (会话临时提升)           │
│  - 所有普通功能                      │
│  - 导出完整数据                      │
│  - 查看全部日志                      │
│  - 管理设备                          │
│  - 管理等级规则                      │
│  - 处理冲突                          │
│  - 运行恢复                          │
│  - 查看审查报告                      │
└─────────────────────────────────────┘
```

### 认证流程

1. 用户点击"管理员入口"按钮
2. 弹出密码输入框
3. 验证管理员密码（Argon2id 派生）
4. 成功后在当前会话中启用管理功能
5. 会话过期或程序关闭后自动退出管理员模式

### 密码存储

- 管理员密码不直接存储
- 使用 Argon2id 派生哈希存储
- 格式：`$argon2id$v=19$m=65536,t=3,p=4$<salt>$<hash>`
- 初始密码通过安全渠道分发

### UI 设计原则

1. **普通模式**：不显示任何管理功能入口
2. **管理员模式**：
   - 顶部显示"管理员模式"标识
   - 管理功能在独立菜单/标签页
   - 关键操作需要二次确认
3. **审计**：所有管理员操作记录到日志

### 多管理员支持

系统支持多个管理员：

- Tom（主管理员）
- 吴汶轩（副管理员）

每个管理员有独立的 AdminId 和密码哈希。

## 数据与兼容性影响

- 管理员信息存储在设备表中
- 密码哈希单独存储，不与事件日志混合

## 安全影响

- 密码必须足够复杂（最小长度 12）
- 登录失败次数限制（5 次/小时）
- 管理员会话超时（30 分钟无操作）
- 所有管理员操作必须记录

## 测试计划

1. 验证普通用户无法访问管理功能
2. 验证管理员密码验证
3. 验证暴力破解防护
4. 验证管理员操作审计日志

## 迁移计划

首次启动时：

1. 生成默认管理员账号
2. 强制要求修改初始密码
3. 记录初始化事件

## 替代方案

1. **独立管理程序**：增加部署复杂度
2. **Web 管理界面**：增加技术栈复杂度
3. **命令行管理**：不便于日常使用

## 未解决问题

- 是否需要管理员双因素认证（Phase 3 考虑）

## 决策

在同一程序中通过管理员密码授权进入管理入口，会话临时提升权限，所有管理员操作必须审计。
